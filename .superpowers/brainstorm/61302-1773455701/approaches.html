3 Architectural Approaches

Replacing the bash daemon — keeping markdown task files

A

Node.js All-in-One Coordinator

One process replaces everything

Human / Web UI
  ↓
company-server (Node.js)
  ├─ fs.watch → task changes → route notifications
  ├─ HTTP :4000 → serves dashboard
  └─ tmux send-keys → wake agents (with retry)
      ↓
Claude agents (tmux windows)
  └─ read/write tasks/*.md

Pros

  • Single process to manage
  • Event-driven (instant response)
  • Web dashboard built-in
  • No polling, no drift

Cons

  • Still relies on tmux for agent wake
B

Node.js + Per-role Inbox Files

Eliminate tmux dependency entirely

Human / Web UI
  ↓
company-server (Node.js)
  ├─ fs.watch → task changes
  ├─ HTTP :4000 → dashboard
  └─ writes roles/<role>/inbox.md
      ↓
Claude agents — CLAUDE.md tells each
  agent to check inbox.md on startup

Pros

  • No tmux dependency
  • Agents pull work, not pushed

Cons

  • Must update all role CLAUDE.md files
  • Agents need manual restart to pick up inbox
C

Node.js + Chokidar + Robust Tmux

Minimal change, maximum reliability

Human / Web UI
  ↓
company-server (Node.js)
  ├─ chokidar → real-time file events
  ├─ HTTP :4000 → dashboard
  └─ tmux send-keys + exponential backoff
      + persistent retry queue
      ↓
Claude agents (tmux windows)

Pros

  • Most robust file watching (chokidar)
  • No CLAUDE.md changes needed
  • Proven tmux pattern, just more reliable

Cons

  • Requires npm install (chokidar dep)
  • Still tmux-coupled

Recommendation

Option A — Node.js all-in-one. Node's built-in fs.watch is event-driven and reliable on macOS. One process to start, monitor, and stop. The tmux coupling is acceptable since agents already run there — the key improvement is replacing bash polling with real-time events and proper error handling.
