Per-Role Dispatch Loop (the key)
Old way (unreliable)
daemon writes task file
↓
tmux send-keys (may fail)
↓
agent reads file... maybe
↓
agent updates status... maybe
↓
daemon polls file change (3s)
↓
race condition possible
New way (reliable)
task goes pending → enters role queue
↓
if role idle: pop task from queue
↓
await sdk.query(taskPrompt, { session_id })
↓           ← agent runs here
ResultMessage received
↓
role DEFINITELY idle now
↓
dispatch next from queue
SDK Hooks Used
TeammateIdle
Fires when agent goes idle — primary idle signal
TaskCompleted
Fires on task completion — update task file
stop_reason
end_turn / max_tokens / refusal — handle errors

What we keep

  • tasks/*.md format (unchanged)
  • YAML frontmatter (unchanged)
  • Role CLAUDE.md files (minor updates)
  • company CLI interface

What we replace

  • bash daemon → Node.js coordinator
  • tmux send-keys → Agent SDK query()
  • file polling → fs.watch events
  • guessed idle state → SDK completion signal
